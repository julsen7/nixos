Um deine Arch Linux-Dotfiles auf NixOS zu übertragen, musst du deine Konfiguration von einem imperativen System (manuell installierte Pakete) in ein deklaratives System (in .nix-Dateien definierter Zielzustand) übersetzen.
Da NixOS die Standard-Verzeichnisstruktur (FHS) bricht, laufen deine alten Skripte und Binärdateien ohne Anpassung meist nicht.
🛠️ Schritt 1: Das Basissystem aufsetzen
Installiere NixOS wie gewohnt über den grafischen Installer (Calamares). Wähle dein Dateisystem und die gewünschte Desktop-Umgebung. Nach dem Reboot generiert NixOS deine Basis-Konfiguration.
Die zentralen Systemdateien liegen unter /etc/nixos/:

    hardware-configuration.nix: Wird automatisch erzeugt (Dateisysteme, Treiber). Nicht manuell ändern.
    configuration.nix: Deine globale Systemkonfiguration (Pakete, Bootloader, Dienste).

⚡ Schritt 4: Die drei größten Arch-zu-NixOS Fallstricke lösen
1. Hardcodierte Pfade in Skripten
In Arch starten Skripte oft mit #!/bin/bash. In NixOS existiert dieser Pfad nicht.

    Lösung: Ändere die Shebangs in deinen Dotfile-Skripten zu #!/usr/bin/env bash.

2. Binärdateien aus AUR / externen Quellen
Heruntergeladene Binärdateien (z.B. per curl installierte CLI-Tools) starten unter NixOS nicht, da der dynamische Linker (ld-linux) an einem anderen Ort liegt.

    Lösung: Nutze das Tool nix-ld in deiner Konfiguration oder verpacke das Tool nativ in Nix.

3. XDG-Verzeichnisse
Wenn deine Arch-Dotfiles fest auf Pfade wie /usr/share/... verweisen, stürzt die Software ab.

    Lösung: Nutze im Home Manager die nativen Programmeinstellungen (z.B. programs.bash.enable = true;), da NixOS die Umgebungsvariablen dann korrekt setzt.

Jede Änderung an deiner Konfiguration aktivierst du mit einem einzigen Befehl:
bash

sudo nixos-rebuild switch

Das ist eine hervorragende und sehr strukturierte Vorlage. Da du bereits UWSM, Hyprland, AMD/Nvidia-Hybrid-Grafik und Home Manager-Konzepte (bisher via stow) nutzt, ist der Umstieg auf NixOS extrem elegant.
Dein gesamtes Bash-Installationsskript wird unter NixOS komplett überflüssig, da NixOS all diese Schritte (Paketlisten einlesen, Symlinks setzen, Systemd-Dienste aktivieren, Dateien patchen) nativ und atomar erledigt.
Hier ist deine vollständige Übersetzung in die deklarative NixOS-Welt, aufgeteilt in die System-Basis (configuration.nix) und deinen Benutzer (home.nix für Home Manager).
💻 1. Die System-Basis (/etc/nixos/configuration.nix)
Diese Datei ersetzt deine Hardware-Treiber, Systempakete, Kernel-Module und globalen systemctl enable Befehle.
nix

{ config, pkgs, ... }:

{
  imports = [ ./hardware-configuration.nix ];

  # 1. Bootloader & Kernel
  boot.loader.systemd-boot.enable = true;
  boot.loader.efi.canTouchEfiVariables = true;
  boot.kernelPackages = pkgs.linuxPackages_latest; # Entspricht Arch 'linux'
  hardware.cpu.amd.updateMicrocode = true;

  # 2. Grafiktreiber (AMD + Nvidia Hybrid-Setup / Prime)
  hardware.graphics = {
    enable = true;
    enable32Bit = true; # Ersetzt lib32-mesa & lib32-vulkan
  };
  
  services.xserver.videoDrivers = [ "nvidia" ];
  
  hardware.nvidia = {
    modesetting.enable = true;
    open = true; # Ersetzt nvidia-open
    nvidiaSettings = true;
    prime = {
      offload.enable = true;
      offload.enableOffloadCmd = true;
      # HINWEIS: Hier musst du die PCI-IDs deiner Hardware eintragen (erfahrbar via 'lspci')
      amdgpuBusId = "PCI:4:0:0"; 
      nvidiaBusId = "PCI:1:0:0";
    };
  };

  # 3. Netzwerk & Sicherheit (Ersetzt ufw, networkmanager)
  networking.networkmanager.enable = true;
  networking.firewall.enable = true; # Integrierte Firewall statt UFW

  # 4. Sound & Firmware (Ersetzt pipewire, sof-firmware, wireplumber)
  security.rtkit.enable = true;
  services.pipewire = {
    enable = true;
    alsa.enable = true;
    alsa.support32Bit = true;
    pulse.enable = true;
    jack.enable = true;
    wireplumber.enable = true;
  };
  hardware.firmware = [ pkgs.sof-firmware ];

  # 5. Bluetooth & Power (Ersetzt bluez, power-profiles-daemon)
  hardware.bluetooth.enable = true;
  services.power-profiles-daemon.enable = true;
  services.udisks2.enable = true;

  # 6. Display Manager & Desktop (Ersetzt ly, hyprland, uwsm, portals)
  services.displayManager.ly = {
    enable = true;
    # Deine Custom-Animation wird direkt hier deklarativ hineingeschrieben:
    settings = {
      animation = "dur_file";
      dur_file_path = "/etc/ly/blackhole.dur";
      bigclock = "en";
    };
  };
  # Schreibt deine Custom-DUR-Datei direkt an den richtigen Ort
  environment.etc."ly/blackhole.dur".source = /home/deinuser/Dotfiles/assets/blackhole.dur;

  programs.hyprland = {
    enable = true;
    withUWSM = true; # Aktiviert UWSM-Integration nativ
  };

Das herauszufinden ist der wichtigste Wendepunkt beim Verstehen von NixOS. In Arch Linux macht ein Paket (pacman -S) einfach alles: Es kopiert die Binärdatei, packt Systemd-Services ins System und legt Standard-Configs ab.
In NixOS ist ein reines Package nur eine isolierte Sammlung von Dateien im schreibgeschützten Nix-Store (/nix/store/...). Es hat keinerlei Berechtigungen, tiefe Systemänderungen vorzunehmen. 
Um herauszufinden, ob ein Programm ein einfaches Paket ist oder einen eigenen Systembefehl benötigt, nutzt man die offizielle Suchmaschine search.nixos.org. 
🔍 Die goldene Regel für die Suche
Auf der Website gibt es oben zwei entscheidende Reiter: Packages und Options. 
1. Wann reicht ein Eintrag unter packages? (Reiter: "Packages")
Wenn ein Programm keine Systemrechte braucht, sich nicht in den Bootloader einreinklinken muss und keine globalen Systemd-Dienste startet. 

    Beispiele: vlc, gimp, btop, kitty, fastfetch.
    Erkennung: Wenn du auf search.nixos.org im Reiter Packages danach suchst, findest du den Paketnamen (z. B. pkgs.gimp). Es gibt dazu keine Systemeinstellungen. Du setzt es einfach auf die Liste. 

2. Wann brauche ich eine programs. oder services. Option? (Reiter: "Options")
Wenn ein Programm tief in das System eingreifen muss. Dazu gehören Grafiktreiber, Display-Manager, Firewalls oder Programme, die spezielle Sandbox-Rechte/Hardware-Zugriffe brauchen.

    Beispiele: steam, hyprland, ufw, docker.
    Erkennung: Schalte auf search.nixos.org auf den Reiter Options um und tippe den Namen ein. Siehst du dort Einträge wie programs.steam.enable oder services.ufw.enable, nutzt du immer diesen Schalter anstelle des reinen Pakets. 

🎮 Das konkrete Beispiel: Warum Steam?
Wenn du auf search.nixos.org nach Steam suchst, findest du sowohl ein Paket (pkgs.steam) als auch eine Option (programs.steam.enable). Warum musst du die Option wählen?
Ein reines Kopieren der Steam-Binärdatei in dein System (environment.systemPackages = [ pkgs.steam ];) schlägt fehl, weil Steam Folgendes benötigt:

    32-Bit Grafikbibliotheken (viele Spiele sind noch 32-Bit, dein System ist 64-Bit).
    Spezielle Firewall-Regeln im System für das Entdecken von lokalen Netzwerken (LAN) und Steam Link.
    Erhöhte Prozess-Prioritäten (Ulimits), damit Spiele flüssig laufen und nicht vom System gekillt werden. 

Ein normales Paket darf unter NixOS diese globalen Systemänderungen nicht vornehmen. Der Befehl programs.steam.enable = true; ist ein sogenanntes NixOS-Modul. Es ist ein vorgefertigtes Skript der Entwickler, das im Hintergrund automatisch das Paket installiert, die 32-Bit-Treiber aktiviert und die Firewall öffnet.

🎯 Warum du nano trotzdem in deine Liste schreiben solltest (Best Practice)
Es ist absolut üblich und empfohlen, deine Lieblingswerkzeuge explizit zu deklarieren, selbst wenn sie scheinbar schon da sind. Dafür gibt es drei wichtige Gründe:
1. Volle deklarative Unabhängigkeit

2. Keine Angst vor doppelter Speicherbelegung
In Arch Linux führt ein doppeltes Installieren zu Warnungen oder Fehlern. In NixOS passiert bei Duplikaten in der Paketliste folgendes: Nichts.

🖥️ Die Monitor-Frage: NixOS-Ebene oder Hyprland-Ebene?
Die klare Antwort lautet: Erst auf Hyprland-Ebene!
1. Warum nicht auf NixOS-Ebene?
Da du Wayland (Hyprland) nutzt, hat das zugrundeliegende Betriebssystem (NixOS) absolut keine Kontrolle über deine Bildschirme, Auflösungen oder Skalierungen. Unter Wayland ist der Window Manager (der Compositor) selbst der Display-Server. NixOS stellt über Grafiktreiber lediglich sicher, dass die Grafikkarte ein Bild ausgeben kann.
2. Wie du es in Hyprland am besten löst
Es gibt zwei bewährte Ansätze, um deine unterschiedlichen Laptop-Bildschirme und externen Monitore zu verwalten:

    # Kleiner Laptop (z.B. eDP-2 ist das interne Display)

Unter NixOS wird alles temporär unterhalb von /mnt zusammengesteckt, damit der Installer weiß, wie das fertige System strukturiert sein soll. Die Reihenfolge ist zwingend einzuhalten: 
```bash
1. Root-Partition nach /mnt mounten
mount /dev/nvme0n1p3 /mnt
2. Den Ordner für die Boot-Partition INNERHALB von /mnt erstellen
mkdir -p /mnt/boot
3. Boot-Partition nach /mnt/boot mounten
mount /dev/nvme0n1p1 /mnt/boot
4. Swap aktivieren (Wichtig, damit der Installer das später erkennt)
swapon /dev/nvme0n1p2
``` 
🧠 Schritt 3: Der NixOS-Zwischenschritt (Hardware erfassen)
Bevor du nixos-install ausführst, musst du NixOS erlauben, deine Hardware (insb. die UUIDs deiner gerade gemounteten Partitionen) zu scannen. Dazu generierst du eine temporäre Basis-Konfiguration direkt in dein gemountetes System: 
bash nixos-generate-config --root /mnt  
Dieser Befehl erstellt den Ordner /mnt/etc/nixos/ und legt dort zwei Dateien ab: 

    hardware-configuration.nix (Enthält die exakten UUIDs deiner SSD-Partitionen).
    configuration.nix (Ein minimales Standard-Skelett). 

📥 Schritt 4: Deine Dotfiles & Flakes ins Spiel bringen
Jetzt überschreibst du das Standard-Skelett mit deiner eigenen Konfiguration, die du auf GitHub vorbereitet hast. Da du ein Flake-basiertes Setup nutzt (wie wir es oben mit den verschiedenen Ordnern für main-laptop und small-laptop besprochen haben), klonst du dein Repo einfach dorthin:
